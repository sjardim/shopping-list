# Deployment

The app supports two deployment topologies. Pick the one that fits.

| Option | Database | Real-time sync | Best for |
|---|---|---|---|
| **A. Laravel Cloud** | PostgreSQL (managed) | Optional via Reverb service | Production with two devices, automated TLS, zero ops |
| **B. Self-hosted VPS** | SQLite (file) | Optional, requires running Reverb yourself | Personal use, single small box, minimal ops |

Real-time sync (the "Lista atualizada" notification when the other user makes changes) requires Reverb. Without it, the app works fully for a single user or for shared use where each person manually refreshes. You can deploy without Reverb and add it later.

The app uses only the Free edition of Flux UI, so there is no proprietary composer repository to authenticate against — `composer install` works for any clone of the repo.

## Option A: Laravel Cloud + PostgreSQL

### Prerequisites

- A Laravel Cloud account
- Git repository connected to Laravel Cloud

## Environment Variables

Set these in the Laravel Cloud dashboard under your project's environment settings.

### Required

```env
APP_NAME="Shopping List"
APP_ENV=production
APP_KEY=                        # generated by artisan key:generate
APP_URL=https://your-domain.com
APP_DEBUG=false

APP_LOCALE=pt_PT
APP_FALLBACK_LOCALE=en

STORES_REGION=pt                 # pt, us, uk, br, or es
CURRENCY_SYMBOL=€                # shown next to every price

# Admin user is created via the database, not env. After deploy run:
#   php artisan lista:install        (interactive: locale, region, currency, admin)
#   php artisan lista:make-admin     (ad-hoc admin creation / promotion)

DB_CONNECTION=pgsql
DB_HOST=                        # provided by Laravel Cloud
DB_PORT=5432
DB_DATABASE=                    # provided by Laravel Cloud
DB_USERNAME=                    # provided by Laravel Cloud
DB_PASSWORD=                    # provided by Laravel Cloud

BROADCAST_CONNECTION=log        # use "reverb" when Reverb is enabled
QUEUE_CONNECTION=database
SESSION_DRIVER=database
CACHE_STORE=database

# Reverb (omit or leave blank to disable real-time sync)
# REVERB_APP_ID=
# REVERB_APP_KEY=
# REVERB_APP_SECRET=
# REVERB_HOST=
# REVERB_PORT=8080
# REVERB_SCHEME=https
# VITE_REVERB_APP_KEY="${REVERB_APP_KEY}"
# VITE_REVERB_HOST="${REVERB_HOST}"
# VITE_REVERB_PORT="${REVERB_PORT}"
# VITE_REVERB_SCHEME="${REVERB_SCHEME}"
```

## Laravel Cloud Setup

### 1. Create the project

1. Go to [cloud.laravel.com](https://cloud.laravel.com) and create a new project.
2. Connect your Git repository.
3. Set the PHP version to **8.3** or higher.

### 2. Provision a database

Add a **PostgreSQL** database from the Cloud dashboard. The `DB_*` variables will be injected automatically.

### 3. Enable Reverb (optional)

Skip this step for the initial deploy. Add it later when you want real-time sync.

1. Add a **Reverb** service from the Cloud dashboard.
2. Set `BROADCAST_CONNECTION=reverb` in environment variables.
3. Set the `REVERB_*` and `VITE_REVERB_*` variables. Cloud provides the `REVERB_HOST` value after provisioning.
4. Rebuild and redeploy so the frontend JS picks up `VITE_REVERB_APP_KEY`.

When `VITE_REVERB_APP_KEY` is absent or empty, the frontend skips the Echo/WebSocket setup entirely — no errors, no reconnect attempts.

### 4. Enable the queue worker (required for Reverb)

Only needed when Reverb is enabled. Add a **Worker** process in the Cloud dashboard:

```
php artisan queue:listen --tries=3 --timeout=60
```

Broadcast events are queued. Without a worker they sit in the database and are never sent.

### 5. Deploy

Push to your connected branch. Cloud will run the build command automatically.

Default build command (set in Cloud dashboard):

```bash
composer install --no-dev --optimize-autoloader && npm ci && npm run build && php artisan migrate --force && php artisan db:seed --force && php artisan config:cache && php artisan route:cache && php artisan view:cache
```

The seeder uses `firstOrCreate`, so re-running it on subsequent deploys is safe.

## Post-Deploy Checklist

- [ ] Visit `/login` and confirm the login page loads.
- [ ] Log in with the email + password you set via `php artisan lista:install` (or the dev defaults `admin@example.com` / `password` if you ran `db:seed` instead).
- [ ] Add an item to the list and confirm it appears.
- [ ] Copy the share URL and open it in a private window — confirm shared mode (no "Finish trip" or "Clear" buttons).
- [ ] (If Reverb is enabled) Toggle an item in one window, confirm the other window shows "Lista atualizada" within a second or two.
- [ ] Finish a trip and confirm the history page shows the archived list.

## Changing the Admin Password

If you ran `db:seed`, the admin account is `admin@example.com` / `password`. Rotate it immediately:

```bash
php artisan tinker --execute 'App\Models\User::admins()->first()->update(["password" => bcrypt("your-new-password")])'
```

If you ran `php artisan lista:install`, the password is whatever you typed during setup — change it the same way, or via `php artisan lista:make-admin <email>` for promotion / re-creation flows.

## Option B: Self-hosted VPS + SQLite

This is the simplest possible deploy: one box, one SQLite file, no Reverb. Real-time sync is disabled. Everything else (history, recipes, prices, prints, exports) works exactly the same.

### Prerequisites

- A VPS or shared host with PHP 8.3+ (8.5 ideal), Composer, Node, and a web server (Nginx or Apache).
- SSH access. No managed database needed.

### Environment variables

```env
APP_NAME="Shopping List"
APP_ENV=production
APP_KEY=                        # generated by artisan key:generate
APP_URL=https://your-domain.com
APP_DEBUG=false

APP_LOCALE=pt_PT
APP_FALLBACK_LOCALE=en

DB_CONNECTION=sqlite
# DB_DATABASE defaults to database/database.sqlite — leave it commented to use the default

STORES_REGION=pt                 # pt, us, uk, br, or es
CURRENCY_SYMBOL=€                # shown next to every price

# Admin user is created via the database, not env. After deploy run:
#   php artisan lista:install        (interactive: locale, region, currency, admin)
#   php artisan lista:make-admin     (ad-hoc admin creation / promotion)

BROADCAST_CONNECTION=log
QUEUE_CONNECTION=database
SESSION_DRIVER=database
CACHE_STORE=database
```

### Steps

1. Clone the repo on the server.
2. `composer install --no-dev --optimize-autoloader`
3. `npm ci && npm run build`
4. `cp .env.example .env`, set `APP_KEY` (`php artisan key:generate`), set `APP_URL`, leave `DB_CONNECTION=sqlite` (it is the default in `.env.example`).
5. `touch database/database.sqlite` and ensure the web server's user has write permission on the file and the surrounding directory.
6. `php artisan migrate --seed --force`
7. `php artisan config:cache route:cache view:cache`
8. Point Nginx/Apache at `public/`.
9. Run `php artisan lista:install` (or `php artisan lista:make-admin <email>` to add an admin without the locale/region prompts).

### Backups

The whole database is one file: `database/database.sqlite`. Back it up with rsync or any cron-driven copy. The app exposes JSON export per list (`/list/{token}/export.json`) for ad-hoc snapshots.

### Adding Reverb later

If you want real-time sync on a self-hosted setup, install Reverb (`composer require laravel/reverb`), follow the standard install steps, run `php artisan reverb:start` under a process supervisor (systemd, supervisor, pm2), and set the `REVERB_*` and `VITE_REVERB_*` env vars. Then rebuild assets so Vite bakes the public key into the JS.

## Local Development

```bash
composer install
cp .env.example .env
php artisan key:generate
touch database/database.sqlite     # uses SQLite by default
php artisan migrate --seed
composer run dev                   # starts server, queue, Vite, Reverb, and Pail together
```

The `composer run dev` command starts all five processes with concurrently:

| Process | Command |
|---|---|
| HTTP server | `php artisan serve` |
| Queue worker | `php artisan queue:listen` |
| Reverb WebSocket | `php artisan reverb:start` |
| Vite asset bundler | `npm run dev` |
| Log viewer | `php artisan pail` |

If you prefer PostgreSQL locally (matching the Laravel Cloud topology), set `DB_CONNECTION=pgsql` in `.env` and provide credentials. The test suite always uses an in-memory SQLite, regardless.
